## Warehouse Management System (Система управления складом)
Warehouse Management System (WMS) — это веб-приложение для автоматизации складского учета, контроля за текущими остатками товаров и ведения прозрачной истории всех складских операций. Проект разработан с упором на надежность, безопасность данных и скорость работы.

В основе системы лежит асинхронная архитектура бэкенда, строгая ролевая модель доступа для сотрудников и администраторов, а также удобный механизм взаимодействия с фронтендом через защищенные сессии и генерацию динамических QR-кодов для маркировки грузов и товаров прямо в браузере.

##  🛠 Стек технологий
`Backend:` Python 3.12, FastAPI, SQLAlchemy 2.0 (AsyncIO), Pydantic v2

`Безопасность и Аутентификация:` * JWT (PyJWT) — JSON Web Tokens для управления сессиями пользователей через защищенные HttpOnly куки.

`Argon2 (argon2-cffi)` — современный криптографический алгоритм хеширования паролей, устойчивый к GPU-перебору и side-channel атакам.

`Database:` PostgreSQL (асинхронный драйвер asyncpg)

`Логирование:` Loguru

`Контейнеризация:` Docker, Docker Compose


## 🗄 Структура базы данных

База данных состоит из 5 основных взаимосвязанных таблиц:

1. **Users (Пользователи)**
* `id` (Primary Key)
* `username` (Unique) — имя пользователя
* `hashed_password` — захешированный пароль
* `role` — роль в системе (`worker`, `admin`, `superadmin`)
* `phone` (Optional) — номер телефона
* `last_login` — время последнего входа


2. **Categories (Категории товаров)**
* `id` (Primary Key)
* `name` (Unique) — название категории
* `description` (Optional) — описание


3. **Products (Товары)**
* `id` (Primary Key)
* `name` — наименование товара
* `sku` (Unique) — артикул товара
* `qr_code_uuid` (Unique) — UUID для генерации QR-кода на фронтенде
* `description` (Optional) — описание товара
* `category_id` (Foreign Key -> Categories) — привязка к категории


4. **Stock (Текущие остатки)**
* `product_id` (Primary Key, Foreign Key -> Products) — связь «один к одному» с товаром
* `quantity` — текущее количество на складе
* `update_at` — дата последнего изменения остатка


5. **Transactions (История операций)**
* `id` (Primary Key)
* `quantity` — количество товара в операции
* `transaction_type` — тип операции (`incoming` — приход, `outgoing` — расход/списание)
* `product_id` (Foreign Key -> Products) — какой товар изменялся
* `user_id` (Foreign Key -> Users) — какой сотрудник совершил операцию
* `created_at` — дата и время проведения транзакции

## 🔌 Структура API

Все эндпоинты разделены по тегам и защищены проверкой ролей через зависимости (Dependencies).

### Пользователи и аутентификация (`/users`)

* `POST /users/login` — Аутентификация пользователя. Генерирует JWT и устанавливает его в безопасную куку `access_token` (`HttpOnly`, `Secure`).
* `POST /users/logout` — Завершение сессии. Бэкенд отправляет инструкцию браузеру на удаление авторизационной куки.
* `GET /users/me` — Получение информации о профиле текущего авторизованного пользователя (Доступ: `worker`, `admin`, `superadmin`).
* `GET /users/` — Получить список всех пользователей системы (Доступ: `admin`, `superadmin`).
* `GET /users/workers` — Получить список всех сотрудников с ролью `worker` (Доступ: `admin`, `superadmin`).
* `GET /users/admins` — Получить список всех администраторов с ролью `admin` (Доступ: `admin`, `superadmin`).
* `GET /users/search` — Поиск пользователя по точному имени `username` (Доступ: `admin`, `superadmin`).
* `POST /users/workers` — Регистрация нового сотрудника (Доступ: `admin`, `superadmin`).
* `POST /users/admins` — Регистрация нового администратора (Доступ: строго `superadmin`).
* `PATCH /users/update_info` — Частичное изменение данных пользователя по `user_id` (Доступ: `admin`, `superadmin`).
* `PATCH /users/change_role` — Изменение роли пользователя по `user_id` (Доступ: строго `superadmin`).
* `DELETE /users/delete_worker` — Удаление сотрудника с ролью `worker` по его `user_id` (Доступ: `admin`, `superadmin`).
* `DELETE /users/delete_admin` — Удаление администратора с ролью `admin` по его `user_id` (Доступ: строго `superadmin`).

### Главная панель (`/dashboard`)

* `GET /dashboard/stats` — Получение агрегированных метрик системы (общее количество товаров, категории и количество товаров с нулевым остатком) без выгрузки тяжелых объектов (Доступ: `admin`, `superadmin`).

### Товары (`/products`)

* `POST /products/create` — Создание нового товара и автоматическая инициализация записи остатков в `Stock` (Доступ: `worker`, `admin`, `superadmin`).
* `GET /products/single` — Получение детальной информации об одном товаре по `product_id` или `name` (Доступ: `worker`, `admin`, `superadmin`).
* `GET /products/` — Получение списка всех товаров с пагинацией, текстовым поиском по названию/SKU, фильтрацией по категориям и гибкой сортировкой (Доступ: `worker`, `admin`, `superadmin`).
* `PATCH /products/{product_id}` — Частичное редактирование данных товара и обновление привязки к категории (Доступ: `worker`, `admin`, `superadmin`).
* `DELETE /products/{product_id}` — Удаление товара из системы по его идентификатору (Доступ: `admin`, `superadmin`).

### Категории (`/categories`)

* `GET /categories/` — Получить список всех существующих категорий (Доступ: `worker`, `admin`, `superadmin`).
* `POST /categories/create` — Создать новую уникальную категорию (Доступ: `worker`, `admin`, `superadmin`).
* `PUT /categories/{cat_id}` — Полное редактирование существующей категории по ID (Доступ: `admin`, `superadmin`).
* `DELETE /categories/{cat_name}` — Удаление категории по её имени. Запрещено, если к категории всё ещё привязаны товары (Доступ: `admin`, `superadmin`).

### Складские транзакции (`/transactions`)

* `POST /transactions/create` — Проведение операции прихода (`arrival`) или расхода (`departure`). Автоматически пересчитывает количество товара на складе и запрещает списания уходящие в минус (Доступ: `worker`, `admin`, `superadmin`).
* `GET /transactions/my` — Просмотр истории транзакций текущего авторизованного пользователя. Поддерживает пагинацию, сортировку и фильтрацию по диапазону дат и типу операции (Доступ: `worker`, `admin`, `superadmin`).
* `GET /transactions/user` — Просмотр истории транзакций любого конкретного сотрудника по его `user_name` (Доступ: `admin`, `superadmin`).
* `GET /transactions/` — Просмотр общей истории абсолютно всех транзакций в системе. Поддерживает фильтрацию по типам `incoming` / `outgoing`, диапазону дат и пагинацию (Доступ: `admin`, `superadmin`).

## 🚀 Как запускать проект

Для локального запуска и развертывания проекта используются файлы конфигурации окружения (`.env`), которые обеспечивают связь между контейнерами Docker и настраивают бэкенд-приложение.

### 1. Корневой `.env` (находится в самом корне проекта)

Используется `docker-compose` для автоматического создания и инициализации контейнера базы данных PostgreSQL.

```env
# Параметры подключения к СУБД для Docker PostgreSQL
DB_USER=postgres
DB_PASSWORD=5225
DB_NAME=warehouse_local

```

### 2. Бэкенд `.env` (находится в папке приложения, рядом с основным кодом)

Содержит полные настройки для работы FastAPI, параметры подключения к БД из кода приложения, секреты JWT-авторизации, а также данные для автоматического создания первого суперпользователя при старте системы.

```env
# Режим отладки приложения
debug=True

# Настройки подключения бэкенда к базе данных
db__DATABASE_HOST=db
db__DATABASE_USER=postgres
db__DATABASE_PASSWORD=5225
db__DATABASE_NAME=warehouse_local

# Учетные данные для автоматического создания первого главного администратора (Superadmin)
db__FIRST_SUPERUSER_USERNAME=superadmin
db__FIRST_SUPERUSER_PASSWORD=1234567
db__FIRST_SUPERUSER_PHONE=+37529678912344

# Конфигурация безопасности и подписи JWT-токенов доступа
jwt__SECRET_KEY=7a5eb579b2e365f5bad0b5fcfb28c25c5f2d0f274d445bca65ab07e8ae52fc14
jwt__algorithm=HS256

```




### Команда для запуска

После настройки обоих файлов конфигурации выполните команду в корне проекта:

```bash
docker-compose up --build

```

После сборки и запуска контейнеров весь проект доступен на стандартном `http://localhost` (порт 80) через настроенный Nginx: фронтенд открывается напрямую в браузере, а все запросы к бэкенду автоматически проксируются через веб-сервер по пути `/api`. При необходимости обратиться к асинхронному API напрямую или открыть интерактивную документацию Swagger (только если debug=True), бэкенд по-прежнему доступен по адресу `http://localhost:8000`.
